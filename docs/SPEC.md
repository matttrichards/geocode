# geo7 Location Code — Specification v1

> A geo7 code turns a precise point on Earth into a short, human-shareable string
> like `CR/h3xex8`, and back, with **no server, no database, and no central
> authority**. This document is the complete, fixed definition. Anyone can
> implement it and obtain bit-identical results.

Status: **frozen — v1 (2026-06-23).** The constants and the region table in this
document are now **immutable** (§9). License: this
specification and the region data (`regions.v1.json`, `js/regions.gen.js`) are
dedicated to the **public domain under CC0 1.0** (`LICENSE-CC0`); the reference
code is licensed under **Apache-2.0** (`LICENSE`) — so anyone may use it for
anything, forever, without permission.

---

## 1. Why it's built this way (design goals)

1. **Permissionless & unstoppable.** The code↔point mapping is a pure function of
   published constants. There is nothing to run, own, or shut down. A third party
   can compute the same codes offline, on paper, or in any language.
2. **Permanent.** A shared code must resolve to the same point forever. This
   requires the inputs to be **frozen** (§9).
3. **Short & human-friendly.** Codes are a few characters in an unambiguous
   alphabet, namespaced by a human-known region prefix (`CR`, `US-CA`) so the
   region costs the *human* no code length — they already know what country
   they're in.
4. **Precise.** ~18 m cells: enough to pinpoint a doorway, coarse enough to keep
   codes short.
5. **Offline.** Encoding/decoding needs only this spec — no network.

There is **no conventional database**: the "registry of every location" is the
mathematical function itself. The only *data* is a small, frozen table of region
bounding boxes (§4).

## 2. Frozen constants

| Constant | Value | Notes |
|---|---|---|
| `ALPHABET` | `0123456789cdefhjkmnpqrtuvwxy` | base-28; index = position (0–27) |
| `BASE` | `28` | `= ALPHABET.length` |
| `TARGET_M` | `18.0` | target cell edge in metres |
| `EARTH_LAT_M_PER_DEG` | `111320.0` | metres per degree latitude |

The alphabet is **digits 0–9 plus 18 letters**, omitting `a b g i l o s z`. The
omissions remove glyphs that are easily confused (`0/o`, `1/i/l`) and trim to a
clean base of 28. Codes are **case-insensitive** (lowercased on decode).

## 3. Region model

A **region** is an axis-aligned WGS84 bounding box `[minLat, maxLat, minLon, maxLon]`
identified by a key:

- Country: ISO-3166-1 alpha-2 — `CR`, `DE`, `JP`, `US`.
- **One box per *contiguous* country** — there are **no political sub-divisions**
  (no states/provinces). A big contiguous country simply uses more characters.
- **Non-contiguous pieces** (a landmass separated by open ocean) get their own box,
  labelled obviously: `US-AK`, `US-HI`, `FR-GF`, `EC-W` (Galápagos), `PT-20`
  (Azores), `RU-CHU` (Chukotka), parent-prefixed ISO territories (`US-PR`,
  `GB-FK`), etc. A couple of non-contiguous countries whose populous part isn't the
  largest landmass are kept whole on purpose (`MY`, `GQ`).

The region key is the **prefix** shown before the slash. A point is encoded
*within* its region's box; the box determines the code length (§5). Region
selection for a raw point uses `detectCountry` (best-effort: prefers a more
specific sub-region, then the smallest containing box); a shared code always
carries its prefix, so detection only matters on the map-tap / GPS convenience
paths.

The full region table — **271 regions, one box per contiguous ISO entity** — is
the frozen data file **`regions.v1.json`** (mirrored as `js/regions.gen.js`),
generated by `tools/gen-regions.mjs` from Natural Earth 1:50m. Re-running the
script on the same input reproduces it byte-for-byte. Each region carries both its
box (`regions`) and its frozen grid integers (`cells` — see §4).

**Provenance (for byte-exact reproduction).** The boxes derive from the
public-domain **Natural Earth 1:50m** vectors `ne_50m_admin_0_countries` and
`ne_50m_admin_1_states_provinces`. The generator pads every box by `MARGIN = 0.03°`,
merges polygons within `MERGE_GAP = 2.0°` into one cluster, breaks out far clusters
beyond `SPLIT_GAP = 5.0°`, and rounds all coordinates to 4 decimal places. The
pinned input checksums are:
```
SHA-256  ne_50m_admin_0_countries.geojson         3e458fc036ad0a66411f2c1e6cac49c5d7bfb81cb1123bc513b22511a2b7fdeb
SHA-256  ne_50m_admin_1_states_provinces.geojson  69a0e06e640b2d505858ae1cb63034e4677f3000b35a98e16312932b98c426b9
```

**Edge cases.** Some boxes intentionally extend slightly past the geographic
limits of ±90° latitude / ±180° longitude (e.g. Antarctica `AQ`, Russia `RU`),
an artifact of padding and the source vertices; this is harmless because points
still fall inside. Longitude is **never wrapped**: antimeridian-crossing countries
are split into separate boxes (`FJ`/`FJ-2`, `RU`/`RU-CHU`, `NZ`/`NZ-3`), so every
box's longitude range is a simple interval.

## 4. Algorithm

Each region carries three **frozen integers** in `regions.v1.json` →
`cells[key] = [latCells, lonCells, numChars]`: the grid dimensions and the code
length. **A conforming implementation MUST read these integers and MUST NOT
recompute them.** This is load-bearing: the only operations that could differ
between platforms are the transcendental functions `cos` and `log` used to *derive*
the counts (their last-bit results are not guaranteed identical across languages).
By freezing the derived integers, the code↔point path uses only exact
integer / IEEE-754 basic arithmetic, so every implementation — in any language,
or on paper — produces bit-identical codes.

**Encode** `(lat, lon)` → code (point must lie inside the box):
```
latCell  = min(latCells-1, floor((lat-minLat)/(maxLat-minLat) * latCells))
lonCell  = min(lonCells-1, floor((lon-minLon)/(maxLon-minLon) * lonCells))
index    = latCell * lonCells + lonCell
code     = base28(index, numChars)        # most-significant char first
```

**Decode** code → point (returns the **cell centre**, the canonical point):
```
index    = base28_to_int(code)
latCell  = floor(index / lonCells)
lonCell  = index % lonCells
lat      = minLat + (latCell + 0.5)/latCells * (maxLat-minLat)
lon      = minLon + (lonCell + 0.5)/lonCells * (maxLon-minLon)
```

**Deriving the frozen counts** (informative — run **once** by `tools/gen-regions.mjs`
to produce `cells`; conforming implementations do *not* run this, they read the
result):
```
refLat   = (minLat <= 0 and maxLat >= 0) ? 0
         : (|minLat| < |maxLat|) ? minLat : maxLat   # box's EQUATOR-MOST latitude
lonM     = 111320 * cos(|refLat| * π/180)            # metres per ° lon — widest here
latCells = max(1, ceil((maxLat-minLat) * 111320 / 18))
lonCells = max(1, ceil((maxLon-minLon) * lonM / 18))
numChars = smallest k ≥ 1 with 28^k ≥ latCells*lonCells   # exact integer form of ceil(log28(total))
```
Longitude is sized at the equator-most latitude (where a degree spans the most
metres), which guarantees every cell is ≤ `TARGET_M` across the whole box.

**Precision** at a region: `latResM = (maxLat-minLat)*111320 / latCells` (and
analogously for lon). Latitude cells are ≈ `TARGET_M` everywhere; longitude cells
are **never coarser than `TARGET_M`** — exactly `TARGET_M` at the equator-most edge
(the guaranteed bound) and finer toward the poles.

Because decode returns the cell centre, every point in a cell encodes to one code,
and that code decodes to one canonical point — round-trip-stable to within the
cell (~18 m). geo7 codes are deliberately *not* infinitely precise; the cell **is**
the unit.

## 5. Code length scales with region size

`numChars = ceil(log28(latCells × lonCells))`. Bigger box ⇒ more cells ⇒ longer
code. Verified test vectors (all ~18 m):

| Region | Example point | Code | Chars |
|---|---|---|---|
| CR (Costa Rica) | San José | `CR/h3xex8` | 6 |
| DE (Germany) | Berlin | `DE/2m8pw3v` | 7 |
| US (United States) | Washington DC | `US/23yuvwf0` | 8 |
| JP (Japan) | Tokyo | `JP/0ncy6pn7` | 8 |

Big *contiguous* countries (US, Russia, Canada, …) reach 8 characters — that is the
accepted cost of "one box per contiguous country, no state splits." Longitude cells
are sized at each box's equator-most latitude (§4), so a larger box only ever costs
more characters, never coarser than 18 m precision.

## 6. Code string format & serialization

A geo7 location is exchanged in one of two equivalent forms.

**Human form** — `REGION/CODE`, e.g. `CR/h3xex8`, `US-AK/0txm1rr2`:

- `REGION` is a region key from the table (§3): an ISO-3166-1 alpha-2 code,
  optionally followed by `-` and an uppercase suffix (`US-AK`, `FR-GF`). Matched
  case-insensitively; **canonical form is uppercase**.
- `/` (U+002F) separates region and code.
- `CODE` is exactly `numChars` characters (§4), each drawn from `ALPHABET` (§2).
  Matched case-insensitively; **canonical form is lowercase**.
- Parsers MUST trim surrounding whitespace and MAY strip internal whitespace
  before parsing. A bare code with no `/` is interpreted against a caller-supplied
  default region.
- A string is **valid iff** the region is known, every code character is in
  `ALPHABET`, and the code length equals that region's `numChars`. Any other
  string is invalid.

**Link form** — a URL query carrying the same data plus an optional note:
```
?c=<REGION>&code=<CODE>[&n=<note>]
```
`c` is the region key, `code` is the bare code, and `n` is an optional free-text
label (clamped to an implementation-defined maximum). The note is **presentational
only** — it is not part of the code and does not affect decoding.

## 8. Conformance & error handling

A conforming implementation:

- reproduces every §7 test vector exactly, using the frozen `regions.v1.json`;
- reads the `cells` integers and never recomputes them (§4);
- treats codes case-insensitively, emitting **lowercase** codes and **uppercase**
  region keys.

Error behavior — the reference implementation throws; other implementations MAY
signal errors idiomatically, but MUST NOT silently emit a code or a point:

- **Encode** of a point outside its region box → error (no clamping to the edge).
- **Encode** or **decode** with an unknown region key → error.
- **Decode** of a string containing a character not in `ALPHABET` → error.
- Decode does **not** itself validate length against `numChars`; a code that is too
  long indexes past the grid and yields a point outside the box. Callers that accept
  untrusted input SHOULD verify `length == numChars` first (§6), as the reference UI
  does.

## 9. Immutability & governance (the load-bearing rule)

A code's meaning must never drift. Therefore, **once v1 is frozen:**

- **`ALPHABET`, `BASE`, `TARGET_M`, every region box, and every region's frozen
  `cells` integers are immutable.** Changing any of them re-points existing codes
  and is forbidden in v1.
- **Additive-only.** New regions may be appended (they can't affect existing
  codes). Appending is the *only* permitted change to the table within a version.
- **Versioning.** Any change to existing constants/boxes requires a new spec
  version (`v2`) with a new namespace; v1 codes keep resolving under v1 forever.
- **Frozen as of v1 (2026-06-23).** The boxes and constants are locked; any future
  change requires a new version (`v2`) under a new namespace, and v1 codes keep
  resolving under v1 forever.

Publishing: `SPEC.md` + `regions.v1.json` + the reference implementations + test
vectors are committed to a public repo under **CC0** (data/spec) and **Apache-2.0**
(code). That, not a server, is what makes geo7 "completely public and completely
fixed."

## 10. Prior art & honest differentiation

geo7 is in the family of open, deterministic geocoding grids. It is worth being
clear-eyed about what already exists:

- **Open Location Code / Plus Codes** (Google, Apache-2.0): open, fixed, global,
  offline, base-20, with a "short code + locality" form (`MQRG+59 Liberia`) that
  mirrors geo7's region prefix. The closest prior art.
- **Geohash** (public domain): base-32 lat/lon interleave; variable precision.
- **S2 / H3**: cell hierarchies for analysis, not human-shareable strings.
- **what3words**: proprietary, closed dictionary — the opposite of this goal.

**geo7's intended differentiators**: a denser base-28 alphabet tuned for
unambiguous human entry; ~18 m cells as the fixed unit; a country/admin **prefix as
free human-supplied entropy** (shorter codes than a global single string); and the
product layer (courier hand-off, notes) rather than the encoding alone. Where geo7
does *not* clearly beat Plus Codes on the math, the moat is the product and the
deliberate, frozen, public-domain simplicity — which should be an explicit choice,
not an accident.
